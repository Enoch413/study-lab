# CODE LAB 내 STUDY LAB 기능 MVP 요구사항 문서 v1.2

## 문서 목적
이 문서는 CODE LAB 웹앱 안에 포함될 우리 학원 내부용 STUDY LAB 기능의 MVP 개발 착수 기준 문서다.  
이번 버전은 학생 입실/퇴실 기록, 공부 시간 계산, 강사용 실시간 현황 확인, 질문 학생의 1:1 이동 흐름까지 포함한 MVP 범위를 확정한다.

## 1. 서비스 한 줄 설명
학생이 CODE LAB 안에서 전체 공부방에 입실해 공부하고, 질문이 있을 때만 강사와 1:1 질문방으로 이동할 수 있는 학원 내부용 자습 기능.

## 2. MVP 범위 한 문장 요약
이번 MVP는 CODE LAB 내부에서 전체 공부방 1개를 기준으로 학생의 입실/퇴실과 오늘 공부 시간을 신뢰성 있게 기록하고, 질문이 있는 학생만 1:1 질문방으로 이동했다가 질문 종료 후 자동 복귀할 수 있는 최소 기능을 만든다.

## 3. 이번 문서에서 확정한 핵심 정책
- 서비스 대상은 우리 학원 내부 사용자만이다.
- STUDY LAB은 독립 서비스가 아니라 CODE LAB 내부 기능이다.
- 서비스 형태는 CODE LAB 웹앱 내부 기능이며 PC 브라우저와 모바일 브라우저를 모두 지원한다.
- MVP에서는 학생에게 방 목록을 보여주지 않는다.
- MVP에서는 학생이 항상 입장하는 전체 공부방 1개를 운영한다.
- 질문 대응을 위해 학생과 강사가 들어가는 1:1 질문방을 시스템이 필요 시 생성하거나 배정할 수 있다.
- 질문방은 학생이 직접 선택하는 일반 방이 아니라 질문 대응용 임시 공간으로 본다.
- 시스템 구조는 향후 다중 자습실 확장을 고려하되, 이번 기본 학습 흐름은 `전체 공부방 1개 + 질문 시 1:1 방 이동` 기준으로 설계한다.
- 로그인만으로 공부가 시작되면 안 된다. 학생이 직접 `입실하기`를 눌러야 공부 세션이 시작된다.
- 학생 1명당 동시에 1개의 활성 공부 세션만 허용한다.
- 오늘 공부 시간 기준은 한국 시간대 `Asia/Seoul`의 00:00:00부터 23:59:59까지다.
- 입실/퇴실 시각과 공부 시간 계산 기준은 모두 서버 시간이다.
- 카메라 상태는 이번 MVP에서 학생의 수동 on/off 표시만 제공한다.
- 카메라 영상 송출, 녹화, 실제 카메라 활성 검증은 제공하지 않는다.
- 전체 공부방에서는 학생 마이크를 항상 음소거 상태로 유지해야 한다.
- 전체 공부방에서는 학생이 마이크를 직접 켤 수 없어야 한다.
- 1:1 질문방에 입장하면 학생과 강사의 마이크는 기본값으로 자동 활성화되어야 한다.
- 질문이 종료되면 학생은 자동으로 전체 공부방으로 복귀해야 하며, 복귀 직후 마이크는 다시 음소거 상태가 되어야 한다.
- CODE LAB에 이미 구현된 로그인/로그아웃 기능을 그대로 사용한다.
- STUDY LAB은 별도의 로그인 화면, 로그아웃 화면, 회원가입 화면을 만들지 않는다.
- 학생 계정은 CODE LAB에 이미 존재하는 계정을 그대로 사용한다.
- 사용자 식별, 로그인 세션, 로그아웃 처리는 CODE LAB의 기존 인증 체계를 따른다.
- STUDY LAB은 CODE LAB의 기존 `role`, `adminScope`, `classIds` 구조를 최대한 재사용한다.
- 강사와 관리자 권한은 CODE LAB의 기존 사용자/권한 체계 안에서 처리하는 것을 기본 전제로 한다.

## 4. 대상 사용자와 권한

| 사용자 | 주요 목적 | 조회 가능 범위 | 수행 가능 행동 |
|---|---|---|---|
| 학생 | 본인 자습 시작/종료, 오늘 공부 시간 확인, 질문 요청 | 본인 정보만 | CODE LAB 로그인 후 입실하기, 퇴실하기, 질문 요청, 카메라 상태 변경, 본인 오늘 기록 조회 |
| 강사 | 현재 자습 현황 확인, 학생별 오늘 공부 시간 확인, 질문 대응 | 전체 학생 정보 | CODE LAB 로그인 후 전체 학생 현황 조회, 질문 학생 1:1 대응 |
| 관리자 | 전체 현황 확인, 운영 관리 | 전체 학생 정보 | CODE LAB 로그인 후 전체 학생 현황 조회, 계정/기본 설정 운영 |

## 4-1. CODE LAB 권한 매핑 기준
- CODE LAB의 기존 사용자 역할 값은 기본적으로 `student`와 `admin`을 사용한다.
- STUDY LAB에서는 별도 `teacher` 역할을 새로 만들지 않는 방향을 우선 검토한다.
- 권장 매핑은 다음과 같다.
- `role = student` 는 학생으로 사용한다.
- `role = admin` 이면서 `adminScope = assigned` 는 강사로 해석한다.
- `role = admin` 이면서 `adminScope = all` 는 관리자로 해석한다.
- 이번 MVP에서는 강사와 관리자 모두 STUDY LAB에서 전체 학생 현황을 조회할 수 있게 한다.
- 향후 담당 반 기준 제한이 필요해지면 CODE LAB의 `classIds`, `adminScope`를 활용해 확장한다.

## 5. 이번 MVP의 운영 기준
- 기본 자습실은 1개다.
- 학생 화면에는 방 선택 UI가 없다.
- 학생은 로그인 후 학생 대시보드에서 바로 `입실하기` 또는 `퇴실하기`를 수행한다.
- 시스템은 기본 자습실 1개에서 학생 100명 동시 입실을 목표로 구현한다.
- `100명`은 운영 목표이자 성능 기준이다.
- 100명을 초과하는 경우의 대기열, 자동 분산, 다중 방 분배 기능은 이번 MVP 범위에서 제외한다.
- 다중 자습실이 필요해질 경우를 대비해 데이터 구조에는 `room_id`를 유지한다.
- 단, 이번 MVP에서는 학생 UI와 강사 UI 모두 방 선택 기능을 노출하지 않는다.

## 6. 필요한 화면 목록
- 학생 대시보드 화면
- 강사/관리자 현황 대시보드 화면
- 1:1 질문방 화면

## 6-1. CODE LAB과의 연결 방식
- STUDY LAB 진입은 CODE LAB 로그인 완료 상태를 전제로 한다.
- 비로그인 사용자가 STUDY LAB URL에 접근하면 CODE LAB의 기존 로그인 흐름으로 이동해야 한다.
- STUDY LAB 화면 상단 또는 공통 레이아웃은 CODE LAB의 기존 공통 구조를 따른다.
- 로그아웃 동작은 STUDY LAB이 별도로 구현하지 않고 CODE LAB 공통 로그아웃 기능을 사용한다.
- CODE LAB 메인 포털 화면에 `STUDY LAB` 진입 카드가 추가되어야 한다.
- CODE LAB 드로어 메뉴의 바로가기 영역에도 `STUDY LAB` 진입 버튼이 추가되어야 한다.
- STUDY LAB은 PREP, CHECK와 동일한 레벨의 모듈로 노출하는 것을 기본안으로 한다.

## 7. 사용자별 주요 행동 흐름

### 학생 흐름
1. 학생은 CODE LAB 계정으로 로그인된 상태에서 STUDY LAB에 진입한다.
2. 학생 대시보드에 진입한다.
3. 현재 활성 공부 세션이 없으면 `입실하기` 버튼이 보인다.
4. 학생이 `입실하기`를 누르면 시스템이 서버 시간으로 입실 세션을 생성한다.
5. 학생은 전체 공부방 입실 상태에서 현재 입실 시각, 오늘 누적 공부 시간, 카메라 on/off 상태를 확인한다.
6. 전체 공부방에서는 학생 마이크가 자동 음소거 상태로 유지된다.
7. 학생은 필요 시 카메라 상태를 on/off로 변경한다.
8. 질문이 생기면 `질문 요청`을 생성한다.
9. 강사가 질문 요청을 수락하면 학생은 1:1 질문방으로 이동한다.
10. 질문방 입장 직후 학생 마이크는 자동으로 켜진다.
11. 질문이 종료되면 학생은 자동으로 전체 공부방으로 복귀한다.
12. 복귀 직후 학생 마이크는 다시 음소거 상태가 된다.
13. 학생이 `퇴실하기`를 누르면 시스템이 서버 시간으로 퇴실 처리한다.
14. 시스템은 오늘 공부 시간을 갱신해 보여준다.

### 강사 흐름
1. 강사는 CODE LAB 로그인 상태에서 STUDY LAB에 진입한다.
2. 강사 대시보드에 진입한다.
3. 전체 학생 목록에서 현재 입실 상태와 오늘 공부 시간을 확인한다.
4. 현재 입실 중인 학생의 입실 시각과 카메라 상태를 확인한다.
5. 질문 요청 대기 학생을 확인한다.
6. 필요 시 학생 이름으로 검색하거나 `입실 중만 보기`로 빠르게 확인한다.
7. 질문 요청을 수락하면 강사는 학생과 함께 1:1 질문방에 입장한다.
8. 질문방 입장 직후 강사 마이크는 자동으로 켜진다.
9. 질문 종료 처리 후 학생의 전체 공부방 자동 복귀를 확인한다.

### 관리자 흐름
1. 관리자는 CODE LAB 로그인 상태에서 STUDY LAB에 진입한다.
2. 관리자도 강사와 동일한 현황 대시보드를 조회할 수 있다.
3. 전체 공부방과 질문방 운영 상태를 점검한다.
4. 계정 발급, 비밀번호 초기화, 기본 설정 변경은 운영자용 내부 방식으로 처리한다.
5. 별도 관리자 백오피스 화면은 이번 MVP 범위에서 제외한다.

## 8. 기능 요구사항

### FR-01. CODE LAB 인증 연동
- STUDY LAB은 CODE LAB의 기존 로그인/로그아웃 기능을 그대로 사용해야 한다.
- STUDY LAB은 별도의 로그인 API를 새로 만들지 않아야 한다.
- STUDY LAB은 CODE LAB의 기존 사용자 세션 또는 인증 토큰을 사용해 현재 사용자를 식별해야 한다.
- 비로그인 사용자가 STUDY LAB에 접근하면 CODE LAB의 기존 로그인 흐름으로 이동해야 한다.
- 로그인 성공 후 STUDY LAB 접근 시 사용자 역할에 따라 기본 진입 화면이 달라야 한다.
- 학생은 학생 대시보드로 이동해야 한다.
- 강사와 관리자는 현황 대시보드로 이동해야 한다.
- 로그아웃 동작은 CODE LAB의 기존 공통 기능을 사용해야 한다.

### FR-01A. CODE LAB 포털 연동
- CODE LAB의 메인 포털 화면에 `STUDY LAB` 카드가 추가되어야 한다.
- 카드 위치와 스타일은 기존 `PREP`, `CHECK`, `ADMIN` 카드 구조와 일관되어야 한다.
- CODE LAB 드로어 메뉴에도 `STUDY LAB` 링크가 추가되어야 한다.
- STUDY LAB 화면에서 `홈` 이동 시 CODE LAB 포털 화면으로 복귀해야 한다.

### FR-02. 학생 대시보드
- 학생은 로그인 후 자신의 현재 상태를 바로 확인할 수 있어야 한다.
- 활성 공부 세션이 없으면 `입실하기` 버튼이 보여야 한다.
- 활성 공부 세션이 있으면 `입실 중` 상태가 보여야 한다.
- 학생은 오늘 누적 공부 시간을 확인할 수 있어야 한다.
- 학생은 오늘 본인의 세션 기록 목록을 확인할 수 있어야 한다.
- 오늘 세션 기록 목록에는 최소한 `입실 시각`, `퇴실 시각`, `종료 상태`가 표시되어야 한다.
- 학생은 입실 중일 때 `질문 요청` 버튼을 볼 수 있어야 한다.
- 학생이 질문방으로 이동 중인지, 질문 중인지, 전체 공부방으로 복귀했는지 상태를 확인할 수 있어야 한다.

### FR-03. 입실 시작
- 학생이 `입실하기`를 누르면 새로운 공부 세션을 시작해야 한다.
- 세션 시작 시각은 서버 시간으로 저장해야 한다.
- 세션 생성 시 전체 공부방의 `room_id`를 함께 저장해야 한다.
- 세션 생성 직후 상태는 `ACTIVE`여야 한다.
- 세션 시작 시 기본 카메라 상태는 `OFF`여야 한다.
- 세션 시작 시 학생 마이크 상태는 `MUTED_LOCKED`여야 한다.
- 이미 `ACTIVE` 상태 세션이 있는 학생은 새 세션을 만들 수 없어야 한다.
- 이 경우 시스템은 기존 활성 세션 정보를 반환하고 기존 세션을 이어서 보여줘야 한다.

### FR-04. 퇴실 종료
- 학생이 `퇴실하기`를 누르면 활성 세션이 종료되어야 한다.
- 종료 시각은 서버 시간으로 저장해야 한다.
- 정상 퇴실 시 세션 상태는 `EXITED`로 저장해야 한다.
- 이미 종료된 세션은 다시 수정되지 않아야 한다.
- 질문방에 있는 학생이 퇴실할 경우 시스템은 먼저 질문방 상태를 종료한 뒤 전체 세션 종료를 처리해야 한다.

### FR-04A. 질문 요청
- 학생은 전체 공부방 입실 중일 때만 `질문 요청`을 생성할 수 있어야 한다.
- 질문 요청 생성 시 요청 시각과 요청 상태를 저장해야 한다.
- 이미 진행 중인 질문 요청이나 질문방 세션이 있는 학생은 중복 질문 요청을 만들 수 없어야 한다.
- 강사는 대기 중인 질문 요청 목록을 확인할 수 있어야 한다.

### FR-04B. 1:1 질문방 이동
- 강사가 질문 요청을 수락하면 시스템은 학생과 강사를 1:1 질문방에 배정해야 한다.
- 질문방은 기존 질문방 재사용 또는 새 질문방 생성 방식 중 하나로 처리할 수 있다.
- 학생이 질문방에 입장하면 현재 위치 상태는 `QUESTION_ROOM`으로 바뀌어야 한다.
- 질문방 입장 직후 학생과 강사의 마이크는 자동으로 켜져야 한다.
- 질문방에서는 학생과 강사만 입장할 수 있어야 한다.

### FR-04C. 질문 종료 및 전체 공부방 자동 복귀
- 강사 또는 시스템이 질문 종료를 처리할 수 있어야 한다.
- 질문이 종료되면 학생은 자동으로 전체 공부방으로 복귀해야 한다.
- 자동 복귀 후 학생 위치 상태는 다시 `MAIN_ROOM`이 되어야 한다.
- 자동 복귀 후 학생 마이크는 다시 `MUTED_LOCKED` 상태가 되어야 한다.
- 질문 종료 시 질문 시작 시각, 종료 시각, 담당 강사 정보를 저장해야 한다.

### FR-05. 현재 접속 상태 판정
- 학생이 입실 중일 때 클라이언트는 60초마다 heartbeat를 서버로 전송해야 한다.
- 서버는 세션별 `last_heartbeat_at`을 저장해야 한다.
- 강사 화면에서 `현재 입실 중`은 `ACTIVE` 상태이면서 최근 heartbeat가 3분 이내인 경우로 정의한다.
- 최근 heartbeat가 3분을 초과하면 서버는 해당 세션을 자동 종료해야 한다.
- 자동 종료된 세션 상태는 `DISCONNECTED`로 저장해야 한다.
- 자동 종료 시 퇴실 시각은 마지막 heartbeat 시각으로 저장해야 한다.

### FR-06. 공부 시간 계산
- 오늘 공부 시간은 오늘 날짜와 겹치는 모든 세션 구간의 합으로 계산해야 한다.
- 계산 기준 날짜는 `Asia/Seoul` 시간대여야 한다.
- `EXITED` 세션은 `퇴실 시각 - 입실 시각`으로 계산해야 한다.
- `DISCONNECTED` 세션은 `마지막 heartbeat 시각 - 입실 시각`으로 계산해야 한다.
- `ACTIVE` 세션은 `현재 서버 시각 - 입실 시각`으로 계산해야 한다.
- 학생과 강사 화면에 표시하는 오늘 공부 시간은 동일한 계산 규칙을 사용해야 한다.
- 공부 시간 계산은 서버에서 수행해야 한다.

### FR-07. 카메라 상태 표시
- 학생은 입실 중일 때만 카메라 상태를 변경할 수 있어야 한다.
- 카메라 상태 값은 `ON` 또는 `OFF` 두 가지다.
- 카메라 상태 변경 시 서버에 즉시 반영되어야 한다.
- 강사와 관리자는 학생의 현재 카메라 상태를 볼 수 있어야 한다.
- 이번 MVP의 카메라 상태는 학생의 수동 상태 표시이며, 실제 카메라 활성 여부를 보장하지 않는다.

### FR-07A. 마이크 상태 정책
- 전체 공부방에서는 학생 마이크 상태를 `MUTED_LOCKED`로 강제해야 한다.
- 전체 공부방에서는 학생이 마이크를 켜는 동작을 할 수 없어야 한다.
- 1:1 질문방에 입장하면 학생과 강사의 마이크를 기본값 `ON`으로 설정해야 한다.
- 1:1 질문방에서의 추가 세부 마이크 제어 정책은 최소 범위로 유지한다.
- 질문 종료 후 전체 공부방으로 복귀하면 학생 마이크는 자동으로 다시 꺼져야 한다.

### FR-08. 강사/관리자 현황 대시보드
- 강사와 관리자는 전체 학생 목록을 조회할 수 있어야 한다.
- 목록에는 최소한 `학생 이름`, `현재 상태`, `입실 시각`, `오늘 공부 시간`, `카메라 상태`, `질문 상태`가 보여야 한다.
- 현재 활성 세션이 없는 학생도 목록에서 조회할 수 있어야 한다.
- 강사는 학생 이름 검색 기능을 사용할 수 있어야 한다.
- 강사는 `입실 중만 보기` 필터를 사용할 수 있어야 한다.
- 강사는 질문 대기 중인 학생과 질문 중인 학생을 구분해서 볼 수 있어야 한다.
- 현황 대시보드는 30초마다 자동 갱신되어야 한다.
- 사용자는 수동 새로고침 없이도 최근 상태를 확인할 수 있어야 한다.

### FR-09. 권한 통제
- 학생은 본인 데이터만 조회할 수 있어야 한다.
- 학생은 다른 학생의 이름, 상태, 공부 시간을 볼 수 없어야 한다.
- 강사와 관리자는 전체 학생 데이터를 조회할 수 있어야 한다.
- 권한 통제는 프론트엔드 숨김만으로 처리하면 안 된다.
- 서버 API에서 역할별 접근 권한을 검사해야 한다.

### FR-10. 계정 운영
- 학생 계정은 CODE LAB에 이미 존재하는 계정을 그대로 사용해야 한다.
- STUDY LAB은 학생 계정을 별도로 생성하거나 복제하지 않아야 한다.
- 강사와 관리자 사용자도 CODE LAB의 기존 사용자 테이블과 권한 체계를 사용해야 한다.
- 강사와 관리자 구분은 CODE LAB의 `role`, `adminScope` 값으로 처리하는 것을 기본안으로 한다.
- 학생용 셀프 회원가입은 제공하지 않는다.
- 학생용 비밀번호 찾기 기능은 STUDY LAB 범위에 포함하지 않는다.
- 비밀번호 초기화는 CODE LAB 또는 운영자의 기존 운영 방식으로 처리한다.
- 이번 MVP에서는 STUDY LAB 전용 계정 관리 화면을 필수 범위에 포함하지 않는다.

### FR-11. 데이터 보존과 이력
- 세션 데이터는 삭제보다 상태 변경 방식으로 관리해야 한다.
- 공부 시간 계산의 기준이 되는 세션 원본 데이터는 보존되어야 한다.
- 세션 종료 상태는 최소 `EXITED`, `DISCONNECTED`를 구분해 저장해야 한다.
- 세션 시작과 종료는 감사 가능한 형태로 서버에 기록되어야 한다.
- 이번 MVP에서는 기록 수동 수정 UI를 제공하지 않는다.
- 질문 요청과 질문방 이동 이력도 감사 가능한 형태로 저장되어야 한다.

### FR-12. 다중 자습실 확장 대비
- 데이터 모델에는 `study_room` 개념을 유지해야 한다.
- MVP 초기 데이터에는 기본 자습실 1개만 생성하면 된다.
- 학생 화면에는 room list나 room selector를 제공하지 않는다.
- 향후 다중 자습실이 필요해져도 기존 세션 데이터와 호환되도록 설계해야 한다.

### FR-13. CODE LAB 사용자 데이터 연동
- STUDY LAB은 학생 이름, 사용자 ID 등 기본 사용자 정보를 CODE LAB의 기존 사용자 데이터에서 조회해야 한다.
- STUDY LAB은 별도의 학생 마스터를 우선 도입하지 않는다.
- STUDY LAB 세션 데이터는 CODE LAB 사용자 ID를 참조해야 한다.
- 권한 판정에 필요한 역할 정보도 CODE LAB의 기존 권한 체계를 우선 사용해야 한다.
- CODE LAB의 현재 사용자 속성 중 `role`, `adminScope`, `classIds`와 충돌하지 않도록 설계해야 한다.
- STUDY LAB 전용 권한 필드 추가는 기존 구조로 표현이 불가능한 경우에만 검토한다.

## 9. 비기능 요구사항
- 반응형 웹이어야 한다.
- CODE LAB의 기존 공통 레이아웃과 인증 흐름을 해치지 않아야 한다.
- 데스크톱 브라우저와 모바일 브라우저에서 핵심 흐름이 동작해야 한다.
- 전체 공부방 1개 기준으로 학생 100명 동시 입실 상황을 처리할 수 있어야 한다.
- 강사 대시보드는 학생 100명 기준에서도 실사용이 가능해야 한다.
- 권한 검사는 서버 기준으로 수행해야 한다.
- 서버와 데이터베이스의 시간 기준은 `Asia/Seoul`로 일관되게 처리해야 한다.
- 모든 주요 기록은 서버에서 생성해야 한다.
- HTTPS 환경을 전제로 한다.
- 전체 공부방의 음성 정책과 질문방의 마이크 자동 활성화 정책이 일관되게 동작해야 한다.

## 11. 성공 기준
- 학생은 로그인 후 1분 이내에 자습 시작 상태까지 진입할 수 있어야 한다.
- 강사는 로그인 후 10초 이내에 현재 입실 학생 현황을 파악할 수 있어야 한다.
- 학생 1명당 활성 세션은 항상 최대 1개여야 한다.
- 서버 기준 시간으로 오늘 공부 시간이 일관되게 계산되어야 한다.
- 학생은 본인 정보만 볼 수 있어야 한다.
- 강사와 관리자는 전체 학생 현황을 볼 수 있어야 한다.
- 학생 100명 동시 입실 상황에서도 핵심 화면 사용이 가능해야 한다.
- 질문 학생은 1:1 질문방으로 이동 후 질문 종료 시 자동으로 전체 공부방으로 복귀해야 한다.
- 전체 공부방에서는 학생 마이크가 켜지지 않아야 한다.

## 12. 오픈 전 확인해야 할 위험요소
- 모바일 브라우저 백그라운드 전환 시 heartbeat 전송이 중단될 수 있다.
- heartbeat 기준 3분 정책이 실제 학원 운영에 적절한지 검증이 필요하다.
- 학생이 다른 기기에서 다시 로그인했을 때 기존 활성 세션을 이어보는 동작을 충분히 테스트해야 한다.
- 학생 100명 기준 강사 대시보드의 로딩 속도와 갱신 부하를 점검해야 한다.
- 카메라 상태를 수동 표시로 운영할 때 강사 기대치와 실제 기능 범위를 사전에 맞춰야 한다.
- CODE LAB의 기존 사용자 식별자와 STUDY LAB 세션 테이블 연결 방식 검토가 필요하다.
- CODE LAB 메인 포털과 드로어에 STUDY LAB 진입 요소를 추가할 때 기존 모바일 UI가 깨지지 않는지 확인이 필요하다.
- 1:1 질문방 오디오 연결이 브라우저 권한과 자동 마이크 활성화 정책 때문에 실패하지 않는지 검증이 필요하다.
- 질문 종료 후 자동 복귀가 지연되거나 실패할 때 복구 정책이 필요하다.

## 14. 개발 메모
- 학생 UI는 `로그인 -> 전체 공부방 입실 -> 질문 시 1:1 이동 -> 자동 복귀 -> 퇴실` 흐름을 우선 구현한다.
- 실제로는 CODE LAB 로그인 완료 후 `학생 대시보드 -> 입실/퇴실 -> 질문 요청` 흐름을 우선 구현한다.
- 강사 UI는 CODE LAB 로그인 완료 후 `현황 대시보드 -> 질문 요청 수락 -> 1:1 질문방 대응` 흐름까지 포함한다.
- CODE LAB 현재 구조상 STUDY LAB은 `PREP`, `CHECK`와 같은 포털 카드형 메뉴로 붙는 것이 가장 자연스럽다.
- room list를 없앴기 때문에 프론트엔드 복잡도는 줄지만, 질문방 이동 로직과 마이크 정책이 새 핵심 품질 요소가 된다.
- 별도 인증 개발을 하지 않기 때문에 세션 상태, heartbeat, 시간 계산, 질문방 이동 로직의 정확도가 MVP 핵심 품질이다.
