# CODE LAB 내 STUDY LAB 2단계 기획/설계 문서 초안

## 문서 목적
이 문서는 CODE LAB 안에 포함될 STUDY LAB 기능의 2단계 기획/설계 초안이다.  
목표는 개발자가 기능 범위, 사용자 흐름, 데이터 기록 기준, 권한 정책을 바로 이해하고 다음 단계인 DB/API/화면 기술 설계로 넘어갈 수 있게 하는 것이다.

## 전제
- STUDY LAB은 독립 서비스가 아니라 CODE LAB 내부 기능이다.
- CODE LAB의 기존 로그인/로그아웃과 사용자 계정을 그대로 사용한다.
- 서비스 대상은 우리 학원 내부 사용자뿐이다.
- 최초 버전은 웹앱으로 시작한다.
- PC 브라우저와 모바일 브라우저에서 모두 사용 가능해야 한다.
- MVP는 최대한 단순하게 시작한다.
- 현재 운영 기준은 학생 약 100명을 수용하는 전체 공부방 1개다.
- 학생과 강사에게 일반 방 목록 선택 UI를 노출하지 않는다.
- 질문이 있는 학생만 강사와 함께 1:1 질문방으로 이동한다.
- 질문이 끝나면 학생은 자동으로 전체 공부방으로 돌아온다.
- 전체 공부방에서는 학생 마이크를 항상 음소거 상태로 유지한다.
- 1:1 질문방에 들어가면 학생과 강사의 마이크는 기본값으로 자동 켜진다.
- 기술적으로 전체 공부방 1개에 학생 100명을 안정적으로 수용하기 어렵다면, 내부적으로 여러 room_id로 분할할 수 있다.
- 다만 내부 분할이 발생하더라도 학생과 강사 경험은 가능한 한 `하나의 STUDY LAB`처럼 유지한다.
- 다중 room 분할은 제품 기능이 아니라 기술적 fallback 수단으로 본다.

## [1. PRD]

### 서비스명
STUDY LAB

### 문제 정의
현재 학원 운영에서는 학생이 온라인 자습에 들어왔는지, 언제 들어왔고 언제 나갔는지, 오늘 얼마나 공부했는지를 한 곳에서 신뢰성 있게 확인하기 어렵다.  
강사는 현재 누가 자습 중인지 빠르게 파악해야 하고, 학생은 본인의 입실 상태와 오늘 공부 시간을 바로 확인할 수 있어야 한다.  
또한 학생이 질문이 있을 때 전체 공부 흐름을 깨지 않고 강사와 1:1로 바로 연결되고, 질문이 끝나면 다시 전체 공부방으로 돌아오는 운영 흐름이 필요하다.  
기존 상용 온라인 자습실 서비스는 기능이 과하거나 비용 구조가 맞지 않을 수 있으므로, 학원 내부 운영에 맞는 단순한 자습 관리 기능이 필요하다.

### 목표
- 학생이 CODE LAB 안에서 전체 공부방에 입장하고 퇴장할 수 있게 한다.
- 입실/퇴실 기록을 서버 기준으로 저장한다.
- 학생별 오늘 공부 시간을 자동 계산한다.
- 강사가 현재 입실 현황과 학생별 오늘 공부 시간을 빠르게 확인할 수 있게 한다.
- 질문이 있는 학생을 1:1 질문방으로 이동시켜 강사가 응대할 수 있게 한다.
- 질문이 끝나면 학생을 자동으로 전체 공부방으로 복귀시킨다.
- 카메라 on/off 상태를 단순 표시해 운영 참고 정보로 활용한다.
- 전체 공부방에서는 학생 마이크를 항상 음소거로 유지한다.
- 1:1 질문방에서는 학생과 강사의 마이크를 자동으로 켠다.
- 기본 운영은 전체 공부방 1개로 단순하게 유지한다.
- 기술적으로 필요할 경우에만 내부 다중 room 분할이 가능하도록 구조를 남긴다.

### 사용자 유형

| 사용자 | 정의 |
|---|---|
| 학생 | CODE LAB 계정으로 로그인해 전체 공부방에 입장하고, 필요 시 질문방으로 이동하는 사용자 |
| 강사 | 학생 전체 현황과 오늘 공부 시간을 확인하고, 질문 학생을 1:1 질문방에서 응대하는 운영 사용자 |
| 관리자 | 운영 설정과 room 정책을 관리하는 사용자 |

### 사용자별 핵심 니즈

| 사용자 | 핵심 니즈 |
|---|---|
| 학생 | 빠르게 입실하고, 전체 공부방에서는 조용히 공부하다가 질문이 생기면 바로 1:1 질문방으로 이동하고 싶다 |
| 강사 | 현재 누가 공부 중인지, 오늘 얼마나 공부했는지 보고, 질문 학생을 바로 1:1로 응대하고 싶다 |
| 관리자 | 운영 구조를 단순하게 유지하면서 전체 공부방과 질문방 정책을 관리하고 싶다 |

- 전체 공부방 퇴장
- 입실 기록 생성
- 퇴실 기록 저장
- 오늘 공부 시간 집계
- 현재 접속 상태 표시
- 카메라 on/off 상태의 단순 표시
- 질문 요청
- 1:1 질문방 이동
- 질문 종료 후 전체 공부방 자동 복귀
- 전체 공부방 마이크 강제 음소거
- 1:1 질문방 마이크 자동 활성화
- 강사용 현재 접속 현황판
- 학생별 오늘 공부 시간 확인
- 내부 다중 방 분할을 고려한 데이터 구조 유지

### 제외 범위
- 학생용 일반 방 목록 화면
- 학생용 일반 방 선택 기능
- 상시 운영용 관리자 방 생성/수정 화면
- 결제
- 채팅
- 화면 공유
- 녹화
- 학부모 앱
- 복잡한 푸시 알림
- 랭킹/게임화
- AI 감독 기능
- 실제 카메라 영상 송출
- 실제 카메라 사용 여부 검증
- 학생 셀프 회원가입

### 성공 기준

| 항목 | 기준 |
|---|---|
| 학생 입실 시작 | 로그인 상태에서 1분 이내 전체 공부방 입실 가능 |
| 강사 확인 속도 | 로그인 후 10초 이내 현재 입실 학생 파악 가능 |
| 세션 정확성 | 학생 1명당 활성 세션 최대 1개 유지 |
| 기록 신뢰성 | 입실/퇴실/공부 시간은 서버 시간 기준으로 계산 |
| 권한 통제 | 학생은 본인 정보만, 강사/관리자는 전체 학생 정보 조회 가능 |
| 운영 성능 | 전체 공부방 1개 기준 학생 100명 동시 입실에서도 핵심 기능 사용 가능 |
| 질문 흐름 | 질문 학생은 1:1 질문방 이동 후 질문 종료 시 자동으로 전체 공부방으로 복귀 가능 |
| 음성 정책 | 전체 공부방에서는 학생 마이크가 켜지지 않고, 질문방에서는 자동으로 켜져야 함 |

## [2. 사용자 흐름]

### 학생 사용자 흐름
CODE LAB 로그인 완료  
→ CODE LAB 메인 포털 진입  
→ `STUDY LAB` 카드 선택  
→ 학생 대시보드 또는 자습실 진입 화면 진입  
→ `입실하기` 클릭  
→ 시스템이 서버 기준 입실 기록 생성  
→ 전체 공부방 입장  
→ 학생 마이크는 자동으로 음소거 상태 고정  
→ 공부 중 상태 표시  
→ heartbeat 전송으로 현재 접속 상태 유지  
→ 학생이 필요 시 카메라 상태 `ON/OFF` 변경  
→ 질문이 생기면 `질문 요청` 클릭  
→ 질문 요청 대기 상태 표시  
→ 강사가 요청 수락  
→ 시스템이 학생을 1:1 질문방으로 이동  
→ 질문방 입장 시 학생 마이크 자동 켜짐  
→ 강사와 질문 진행  
→ 질문 종료  
→ 시스템이 학생을 전체 공부방으로 자동 복귀  
→ 복귀 직후 학생 마이크 자동 음소거  
→ `퇴실하기` 클릭  
→ 시스템이 서버 기준 퇴실 기록 저장  
→ 오늘 공부 시간 재계산  
→ 학생 대시보드에서 오늘 공부 시간과 오늘 세션 기록 확인

### 강사 사용자 흐름
CODE LAB 로그인 완료  
→ CODE LAB 메인 포털 진입  
→ `STUDY LAB` 카드 선택  
→ 강사 대시보드 진입  
→ 전체 학생 현황 목록 조회  
→ 현재 입실 여부, 입실 시각, 오늘 공부 시간, 카메라 상태 확인  
→ 질문 요청 대기 학생 확인  
→ 질문 요청 수락  
→ 시스템이 강사를 1:1 질문방에 입장시킴  
→ 질문방 입장 시 강사 마이크 자동 켜짐  
→ 학생과 1:1 질문 진행  
→ 질문 종료 처리  
→ 학생 자동 복귀 확인  
→ 필요 시 학생 이름 검색  
→ 필요 시 `입실 중만 보기` 필터 적용  
→ 특정 학생의 오늘 공부 상태 확인  
→ 필요 시 새로고침 또는 자동 갱신 결과 확인

### 관리자 사용자 흐름
CODE LAB 로그인 완료  
→ CODE LAB 메인 포털 진입  
→ `STUDY LAB` 카드 선택  
→ 관리자 대시보드 진입  
→ 강사 대시보드 수준의 학생 현황 확인  
→ 전체 공부방과 질문방 운영 상태 확인  
→ 운영에 필요한 기본 설정 점검

### 기술 fallback 흐름
운영 중 전체 공부방 1개로 학생 100명을 안정적으로 수용하기 어렵다고 판단됨  
→ 운영자 또는 개발자가 내부 설정에서 room을 여러 개로 분할  
→ 학생은 여전히 `입실하기`만 수행  
→ 시스템이 내부 규칙에 따라 room_id를 자동 배정  
→ 질문 요청과 1:1 질문방 흐름은 동일하게 유지  
→ 강사 화면은 필요 시 내부 room_id를 숨기거나 보조 정보로만 표시

- STUDY LAB은 CODE LAB의 기존 `role`, `adminScope`, `classIds` 구조를 우선 활용한다.

### 내부 room 분할 정책 메모
- 전체 공부방 1개를 기본 전제로 한다.
- 질문방은 학생 선택 대상이 아니라 질문 요청 수락 시 시스템이 자동 배정하는 room이다.
- 기술적으로 100명 수용이 불안정할 때만 여러 room_id를 생성한다.
- 내부 분할 room은 학생 선택 대상이 아니라 시스템 자동 배정 대상이다.
- 강사 화면에서는 필요 시 room 이름을 숨기고 전체 입실 현황만 보여줄 수 있다.

- 질문방 입장과 복귀 시 마이크 상태 변경 이력을 남길 수 있어야 한다.

## 확정 필요 항목
- CODE LAB 내 권한 매핑을 `student / admin+assigned / admin+all`로 확정할지 여부
- 학생에게 room 선택 UI를 완전히 제거하고 바로 입실 흐름으로 고정할지 여부
- 관리자 방 관리 화면을 기본 비노출로 두고, 내부 room 분할이 필요할 때만 열지 여부
- 강사의 조회 범위를 전체 학생으로 둘지, 향후 담당 반 기준으로 제한할지 여부
- 기본 방 최대 수용 인원 표시를 실제 제어값으로 쓸지, 단순 표시값으로 쓸지 여부
- 연결 끊김 자동 종료 기준을 `마지막 heartbeat 후 3분`으로 확정할지 여부
- 강사 대시보드 자동 갱신 주기를 `30초`로 확정할지 여부
- 공부 시간 집계 결과를 실시간 계산할지, 별도 집계 캐시를 둘지 여부
- 카메라 상태를 학생 수동 토글만 허용할지 여부
- 세션 종료 사유를 어느 수준까지 저장할지 여부
- 질문 요청을 학생이 직접 취소할 수 있게 할지 여부
- 질문방에서 강사와 학생이 수동으로 마이크를 다시 끌 수 있게 할지 여부

## 추가 요청 1. 개발자가 바로 질문할 가능성이 높은 애매한 부분 10개
1. 학생은 질문 요청 전에 반드시 전체 공부방 입실 상태여야 하나
2. 질문 요청이 여러 개 동시에 들어오면 어떤 순서로 처리하나
3. 질문 요청은 학생이 직접 취소할 수 있나
4. 강사가 질문 요청을 수락한 뒤 학생 연결이 끊기면 어떻게 처리하나
5. 질문방은 매번 새로 만들지, 기존 질문방을 재사용할지
6. 질문방 입장 시 마이크 자동 켜기가 브라우저 정책 때문에 실패하면 어떻게 처리하나
7. 질문 종료는 강사만 할 수 있나, 학생도 할 수 있나
8. 질문방 시간이 오늘 공부 시간에 포함되나
9. 전체 공부방에서 학생 마이크를 UI와 서버 모두에서 강제 차단할 것인가
10. 내부 room 분할이 생긴 경우 강사 화면에서 room 구분을 보여줄 것인가

## 추가 요청 2. 그 애매한 부분에 대한 권장 기본안
1. 질문 요청은 전체 공부방 입실 중일 때만 가능하게 한다
2. 질문 요청은 먼저 들어온 순서대로 강사 화면에 보여준다
3. 학생은 강사가 수락하기 전까지만 질문 요청을 취소할 수 있게 한다
4. 강사 수락 후 학생 연결이 끊기면 질문 요청 상태를 `CANCELED` 또는 `FAILED`로 종료한다
5. 질문방은 구현 단순화를 위해 고정된 소수의 1:1 room을 재사용하거나, 가상 room 세션 개념으로 처리한다
6. 마이크 자동 켜기가 실패하면 사용자에게 권한 허용 안내를 보여주고 수동 재시도를 유도한다
7. 질문 종료는 기본적으로 강사가 처리하되, 예외적으로 시스템 타임아웃 종료를 허용한다
8. 질문방 시간도 학생의 오늘 공부 시간에 포함한다
9. 전체 공부방에서는 학생 마이크를 UI와 상태 정책 양쪽에서 강제 차단한다
10. 내부 room 분할이 생겨도 강사 기본 화면은 전체 학생 현황 중심으로 유지하고 room 정보는 보조값으로만 쓴다

## 추가 요청 4. 다음 단계에서 바로 작성할 DB 스키마 초안의 준비 메모
