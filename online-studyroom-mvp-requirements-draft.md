# 우리 학원 전용 온라인 자습실 웹앱 요구사항 정리 문서 초안

## 문서 목적
이 문서는 우리 학원 내부 운영용 온라인 자습실 웹앱의 1차 MVP 범위를 정리한 초안이다.  
목표는 학생 입실 관리, 강사 현황 확인, 공부 시간 기록이라는 핵심 운영 기능만 먼저 빠르게 구현하는 것이다.

## 1. 서비스 한 줄 설명
학원 학생이 온라인 자습방에 입실해 공부하고, 강사와 관리자가 실시간 접속 현황과 공부 시간을 확인할 수 있는 내부용 웹앱.

## 2. 서비스 목적
- 학생이 정해진 온라인 자습방에 쉽게 들어와 공부를 시작할 수 있어야 한다.
- 강사가 현재 어떤 학생이 자습 중인지 즉시 확인할 수 있어야 한다.
- 학생별 입실, 퇴실, 총 공부 시간을 신뢰성 있게 기록할 수 있어야 한다.
- 학원 내부 운영에 필요한 최소 기능만 우선 구현해 단순하게 시작해야 한다.
- PC와 모바일 브라우저에서 모두 접속 가능하도록 웹앱 형태로 시작해야 한다.

## 3. 대상 사용자별 사용 목적

### 3-1. 학생
- 본인 계정으로 로그인한다.
- 오늘 참여 가능한 스터디룸 목록을 확인한다.
- 스터디룸에 입장하고 자습 상태를 유지한다.
- 본인의 오늘 입실 기록, 퇴실 기록, 총 공부 시간을 확인한다.
- 자습 중 카메라 상태를 on/off로 표시한다.

### 3-2. 강사
- 강사 계정으로 로그인한다.
- 현재 어떤 학생이 어떤 방에서 공부 중인지 확인한다.
- 학생별 오늘 공부 시간과 현재 입실 상태를 확인한다.
- 학생의 카메라 on/off 상태를 확인한다.

### 3-3. 관리자(원장 또는 운영 담당자)
- 방 생성, 수정, 비활성화 등 스터디룸 운영 설정을 관리한다.
- 사용자 권한 구조를 관리한다.
- 전체 운영 데이터가 정상적으로 기록되는지 점검한다.

## 4. 꼭 필요한 핵심 기능

### 4-1. 로그인 및 권한 구분
- 학생과 강사는 로그인할 수 있어야 한다.
- 로그인 후 사용자 권한에 따라 다른 화면과 기능이 보여야 한다.
- 학생은 자기 정보만 조회할 수 있어야 한다.
- 강사는 학생 전체 현황을 조회할 수 있어야 한다.
- 관리자만 스터디룸 생성, 수정, 비활성화 권한을 가져야 한다.

### 4-2. 스터디룸 목록 조회
- 학생은 입장 가능한 스터디룸 목록을 볼 수 있어야 한다.
- 각 스터디룸에 대해 최소한 방 이름, 운영 상태, 현재 입장 가능 여부를 확인할 수 있어야 한다.
- 비활성화된 방은 학생에게 입장 불가 상태로 표시되어야 한다.

### 4-3. 학생의 스터디룸 입장 및 퇴장
- 학생은 스터디룸에 입장할 수 있어야 한다.
- 입장 시 입실 시간이 자동 기록되어야 한다.
- 학생은 자습 종료 시 퇴실할 수 있어야 한다.
- 퇴실 시 퇴실 시간이 자동 기록되어야 한다.
- 한 학생이 동시에 여러 방에 중복 입실되지 않도록 제한해야 한다.

### 4-4. 입실/퇴실 기록 저장
- 입실/퇴실 시각은 사용자 기기 시간이 아니라 서버 기준 시간으로 기록되어야 한다.
- 학생의 입실, 퇴실, 방 정보, 세션 상태를 저장해야 한다.
- 비정상 종료 상황도 추적할 수 있어야 한다.
  - 예: 브라우저 종료, 새로고침, 네트워크 끊김
- 기록은 사후 수정이 어렵도록 서버에서 생성, 저장, 계산되어야 한다.

### 4-5. 공부 시간 계산
- 학생별 오늘 공부 시간을 자동 계산해야 한다.
- 공부 시간은 서버에 저장된 입실/퇴실 기록을 기준으로 계산해야 한다.
- 하루 기준 총 공부 시간은 누적 합계로 보여야 한다.
- 비정상 종료나 중복 세션 발생 시에도 계산 오류를 최소화할 수 있어야 한다.

### 4-6. 강사용 현재 접속 현황 확인
- 강사는 현재 입실 중인 학생 목록을 볼 수 있어야 한다.
- 학생별로 최소한 다음 정보를 확인할 수 있어야 한다.
  - 학생 이름
  - 현재 입실 여부
  - 입실한 방 이름
  - 입실 시각
  - 오늘 누적 공부 시간
  - 카메라 on/off 상태
- 강사는 목록을 빠르게 확인할 수 있어야 하며, 모바일에서도 주요 정보가 보여야 한다.

### 4-7. 학생별 오늘 공부 시간 확인
- 학생은 본인의 오늘 공부 시간을 확인할 수 있어야 한다.
- 강사는 학생 전체의 오늘 공부 시간을 확인할 수 있어야 한다.
- 관리자는 필요 시 전체 현황을 확인할 수 있어야 한다.

### 4-8. 카메라 on/off 상태 표시
- 학생은 자습 중 카메라 상태를 on/off로 표시할 수 있어야 한다.
- 강사는 학생별 카메라 상태를 볼 수 있어야 한다.
- 이번 MVP에서는 카메라 영상 송출, 녹화, 저장 기능은 제공하지 않는다.
- 카메라 상태는 단순 운영 표시용이며, 감독 증빙 수단으로 과도하게 해석하지 않는다.

### 4-9. 관리자용 방 관리
- 관리자는 스터디룸을 생성할 수 있어야 한다.
- 관리자는 방 이름, 사용 여부, 기본 설명 등의 최소 정보를 수정할 수 있어야 한다.
- 사용 종료된 방은 삭제보다 비활성화 처리할 수 있어야 한다.

### 4-10. 확장 가능성을 고려한 기본 구조
- 사용자, 방, 입실 세션, 일별 공부 시간 데이터를 분리해 관리할 수 있어야 한다.
- 향후 출결 통계, 알림, 학부모 공유 기능을 붙일 수 있도록 구조를 단순하지만 확장 가능하게 설계해야 한다.

## 5. 이번 MVP에서 제외할 기능
- 결제 기능
- 학생 간 또는 학생-강사 간 채팅
- 화면 공유
- 화상 수업 기능
- 녹화 및 다시보기
- 학부모용 앱 또는 학부모 전용 화면
- 복잡한 푸시 알림, 문자, 카카오 알림 시스템
- 랭킹, 배지, 게임화 요소
- AI 감독, 부정행위 탐지, 집중도 분석
- 세부 통계 리포트, 고급 분석 대시보드
- 외부 고객 회원가입 및 공개 서비스 기능

## 6. 필요한 화면 목록
- 로그인 화면
  - 학생/강사 로그인
- 학생용 스터디룸 목록 화면
  - 입장 가능한 방 목록
- 학생용 자습방 입장 상태 화면
  - 현재 방 정보, 입실 상태, 입실 시각, 카메라 on/off 상태, 퇴실 버튼
- 학생용 오늘 공부 시간 확인 화면
  - 본인 오늘 누적 공부 시간, 최근 입실/퇴실 기록
- 강사용 현재 접속 현황 화면
  - 현재 입실 학생 목록, 방 정보, 오늘 공부 시간, 카메라 상태
- 관리자용 스터디룸 관리 화면
  - 방 생성, 수정, 비활성화

## 7. 사용자별 주요 행동 흐름

### 7-1. 학생 흐름
1. 학생이 로그인한다.
2. 스터디룸 목록을 확인한다.
3. 원하는 스터디룸에 입장한다.
4. 시스템이 입실 시간을 서버 기준으로 기록한다.
5. 학생은 자습 중 카메라 상태를 on/off로 표시한다.
6. 학생이 퇴실 버튼을 누르거나 세션 종료 조건이 발생한다.
7. 시스템이 퇴실 시간을 기록하고 오늘 공부 시간을 갱신한다.
8. 학생은 본인의 오늘 공부 시간을 확인한다.

### 7-2. 강사 흐름
1. 강사가 로그인한다.
2. 현재 접속 현황 화면에 들어간다.
3. 현재 입실 중인 학생과 방 정보를 확인한다.
4. 학생별 오늘 공부 시간과 카메라 상태를 확인한다.
5. 필요 시 운영상 조치를 한다.
  - 예: 미입실 학생 확인, 카메라 off 학생 확인

### 7-3. 관리자 흐름
1. 관리자가 로그인한다.
2. 스터디룸 관리 화면에 들어간다.
3. 방을 생성, 수정, 비활성화한다.
4. 운영에 필요한 기본 설정이 올바른지 점검한다.

## 8. 성공 기준
- 학생이 로그인 후 1분 이내에 스터디룸 입장까지 완료할 수 있어야 한다.
- 강사는 현재 누가 공부 중인지 10초 이내에 파악할 수 있어야 한다.
- 학생별 오늘 공부 시간이 운영상 신뢰 가능한 수준으로 기록되어야 한다.
- 학생은 본인 정보만 조회되고, 강사는 학생 전체 현황을 조회할 수 있어야 한다.
- 관리자 외 사용자는 방 생성/수정 기능에 접근할 수 없어야 한다.
- PC와 모바일 브라우저에서 핵심 흐름이 모두 동작해야 한다.
- 운영 초기 기준으로 입실/퇴실 기록 누락률이 매우 낮아야 한다.

## 9. 오픈 전 확인해야 할 위험요소
- 브라우저를 그냥 닫았을 때 퇴실 처리 정책이 불명확하면 공부 시간 오차가 커질 수 있다.
- 한 학생이 여러 기기에서 동시에 로그인할 경우 중복 세션 문제가 발생할 수 있다.
- 카메라 상태를 단순 표시만 할 경우 실제 감독 수준에 대한 운영 기대치 차이가 생길 수 있다.
- 모바일 브라우저에서 백그라운드 전환 시 세션 유지 방식이 불안정할 수 있다.
- 네트워크 불안정 시 입실 상태와 실제 사용자 상태가 다르게 보일 수 있다.
- 권한 분리가 부정확하면 학생 정보 노출 문제가 생길 수 있다.
- 서버 시간 기준이 일관되지 않으면 오늘 공부 시간 계산에 오류가 생길 수 있다.
- 운영자가 원하는 출결 기준과 실제 기록 기준이 다르면 분쟁이 생길 수 있다.

## 10. 추가로 기획자가 결정해야 할 사항
- 학생과 강사 계정은 누가, 어떤 방식으로 생성할 것인지
- 로그인 방식은 아이디/비밀번호인지, 학원 관리번호 기반인지
- 스터디룸은 자유 입장인지, 학생별 지정 방만 입장 가능한지
- 학생이 동시에 여러 기기에서 접속했을 때 차단할지 허용할지
- 브라우저 종료, 네트워크 끊김, 장시간 무응답 시 자동 퇴실 처리 기준
- 카메라 on/off 상태를 학생 수동 토글로 할지, 브라우저 권한 상태와 연동할지
- 오늘 공부 시간 기준을 자정 기준으로 할지, 학원 운영일 기준으로 할지
- 강사가 볼 수 있는 학생 범위를 전체로 할지, 담당 반/담당 학생으로 제한할지
- 과거 기록 조회 범위를 MVP에 포함할지, 오늘 기록만 우선 제공할지
- 관리자 외에 운영 담당자에게 일부 권한을 추가로 줄지

## 이 요구사항 기준으로 다음 단계에서 설계해야 할 것
- 사용자 권한 체계 설계
  - 학생, 강사, 관리자 권한별 조회/수정 가능 범위 정의
- 입실 세션 처리 정책 설계
  - 입실 시작, 퇴실 종료, 비정상 종료, 자동 종료 규칙
- 공부 시간 계산 규칙 설계
  - 하루 기준, 중복 세션 처리, 예외 상황 정산 방식
- 데이터 구조 설계
  - 사용자, 스터디룸, 입실 세션, 일별 합계, 카메라 상태 로그
- 화면 흐름 및 정보 구조 설계
  - 학생/강사/관리자별 핵심 화면과 이동 구조
- API 설계
  - 로그인, 방 조회, 입장, 퇴실, 현재 현황 조회, 오늘 공부 시간 조회
- 운영 정책 설계
  - 계정 발급, 권한 부여, 기록 정정, 장애 대응
- 배포 및 운영 환경 설계
  - 내부용 도메인, 관리자 접근 방식, 로그 보관 기준

## 부록 1. MVP 범위 한 문장 요약
우리 학원 내부에서 학생의 온라인 자습방 입실/퇴실과 공부 시간을 기록하고, 강사가 실시간 현황을 확인할 수 있는 최소 기능의 웹앱을 1차 MVP 범위로 한다.

## 부록 2. 개발 전에 반드시 확정해야 할 정책 5개
1. 로그인 계정 발급 및 초기 비밀번호 관리 정책
